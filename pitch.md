# DS Adoption Scanner — зачем нам это нужно

> RFC-документ для продактов и лидов продуктовой аналитики

---

## Проблема

Дизайн-система внедряется месяцами. Команда DS тратит ресурсы на компоненты, документацию, поддержку — но никто не знает, насколько это реально используется в продуктах.

Типичная ситуация:
- Ручной аудит: 1–2 недели работы, устаревает сразу после публикации
- Нет единого числа → нет цели → нет ответственности
- DS-команда не может доказать ROI своей работы
- Продуктовые команды не знают, что уже готово в DS — и пишут кастомные компоненты, которые дублируют существующие

---

## Решение

**DS Adoption Scanner** — CLI-инструмент, который за несколько минут сканирует все React/TypeScript репозитории компании и выдаёт точные цифры по DS-adoption.

### Как это работает

Инструмент разбирает каждый файл через синтаксическое дерево кода (AST), а не просто ищет текст. Это значит, что он понимает реальные импорты — в том числе через псевдонимы путей, barrel-экспорты и вложенные re-exports. Каждый JSX-компонент в коде получает одну из категорий:

| Категория | Что это | Учитывается в adoption |
|-----------|---------|------------------------|
| **DS-компонент** | Импорт из пакетов дизайн-системы | ✅ числитель |
| **Shared-библиотека** | Внутренние shared/ui, @company/components | ✅ знаменатель |
| **Переиспользуемый кастомный** | Локальный компонент, используется в ≥ 2 файлах | ✅ знаменатель |
| **Уникальный кастомный** | Один файл — один компонент (страничные виджеты) | ✅/❌ настраивается |
| **Third-party** | formik, react-select, react-dnd | ❌ исключён |
| **HTML-нативный** | `<div>`, `<span>`, `<button>` | ❌ исключён |

**Формула adoption:**
```
Adoption Rate = DS-компоненты / (DS + shared-lib + кастомные) × 100
```

### Эффективный adoption

Многие компании используют обёртки над DS — например, внутренний `@company/pro-table` построен поверх DS-таблицы. Прямой счёт покажет, что `ProTable` — не DS-компонент. Но инструмент умеет сканировать исходники таких библиотек и определять DS-backing на уровне каждого компонента:

```
Эффективный Adoption Rate = (DS + транзитивные) / расширенный знаменатель × 100
```

Это даёт **справедливую оценку** — не занижает реальное использование DS через обёртки.

## Что дают цифры бизнесу

### 1. KPI для DS-команды
Adoption Rate — объективный показатель, который можно ставить как цель квартала. Текущий: 42% → цель Q3: 60%.

### 2. Доказательство ROI
История сканов показывает динамику: +8% adoption за квартал = DS-компоненты вытеснили 8% кастомного кода. Можно посчитать в часах разработки.

### 3. Gap-анализ по семьям компонентов
Инструмент группирует DS-компоненты в семьи (Button, Form, Table, Modal...) и показывает, какие семьи не используются вовсе. Это сигнал: либо компонент не нужен, либо команды о нём не знают, либо он уступает кастомному решению.

```
DS Catalog Coverage:  9 / 15 семей = 60%
Не используются: DatePicker, Tree, Transfer, Skeleton, Tour
```

### 4. Дублирование кода
Раздел Local Families показывает группы кастомных компонентов, которые разные команды написали самостоятельно:

```
containers   — 12 instances в 8 файлах (3 компонента)
AlertBar     — 6 instances в 6 файлах
PageHeader   — 4 instances в 4 файлах
```

Если `AlertBar` есть в 6 файлах разных продуктов — это кандидат либо на перенос в DS, либо уже есть аналог в DS, который не нашли.

### 5. Приоритизация DS-роадмапа
Компоненты с высоким кастомным usage — первые кандидаты на добавление в DS. Не нужно угадывать, что нужно командам: данные показывают это прямо.

## Точность и погрешность

Инструмент использует синтаксический анализ кода (AST) и TypeScript API для резолва импортов — это принципиально точнее поиска по тексту.

**Что считается точно:**
- Все статические JSX-импорты, включая barrel-экспорты и path aliases
- Транзитивные компоненты: per-компонентная проверка исходников библиотеки (бинарно: 0 или 1, без усреднений)

**Что намеренно не считается:**
- Динамические импорты (`React.lazy(() => import(...))`) — runtime, не статический анализ
- Компоненты, переданные как строка (`React.createElement('MyButton')`) — редкие паттерны
- Пакеты, не указанные в конфиге `designSystems[]` — намеренное ограничение области

**Главный источник погрешности** — неполный конфиг: если DS-пакет не указан, его компоненты уйдут в `third-party`. Это видно в отчёте и легко исправляется.

**Итоговая точность для типичного React/TypeScript проекта: ~95–98%** от реального DS-использования.

## Сравнение с альтернативами

| | DS Adoption Scanner | omlet.dev | react-scanner |
|---|---|---|---|
| **Хостинг** | Self-hosted | SaaS (данные в облако) | Self-hosted |
| **Стоимость** | Open-source | Платный | Open-source |
| **Adoption rate с формулой** | ✅ | ❌ | ❌ |
| **Транзитивный adoption** | ✅ per-компонент | ❌ | ❌ |
| **Семьи / gap-анализ** | ✅ | ❌ | ❌ |
| **История и динамика** | ✅ | ✅ | ❌ |
| **AI-инструкции для агентов** | ✅ | ❌ | ❌ |
| **Multi-repo за один запуск** | ✅ | ✅ | ❌ |
| **Конфигурация** | 1 ts-файл | SDK в каждом репо | JSON-конфиг |
| **Данные покидают инфраструктуру** | ❌ | ✅ | ❌ |

**omlet.dev** — хороший инструмент для вопроса «кто использует какой компонент», но не отвечает на вопрос «какой % кодовой базы на DS». Требует встроить SDK в каждый репозиторий. Данные уходят в облако (актуально для компаний с ограничениями на передачу кода).

**react-scanner** — простой счётчик: сколько раз использован каждый компонент из пакета. Нет категоризации, нет формулы adoption, нет истории, нет семей, нет понятия shared-библиотек.

## Как внедрить

### Первый запуск (~30 минут)

```bash
npm install -g ds-adoption-scanner
ds-scanner init          # создаёт .ds-scanner.config.ts
# отредактировать: добавить репозитории и пакеты DS
ds-scanner analyze       # первый скан → ds-report.json + ds-report.html
```

Отчёт сразу видно в браузере. JSON-файл можно передать AI-агенту для детального анализа.

### CI/CD интеграция

```yaml
# Одна строка в GitHub Actions / GitLab CI
- run: npx ds-adoption-scanner analyze --save-history --min-adoption 60
```

При падении ниже порога — exit code 1, pipeline падает. История накапливается автоматически.

### Настройка знаменателя

Если в продуктах много страничных one-off компонентов, которые не планируются к замене DS — их можно исключить из знаменателя, чтобы метрика отражала только реально конкурентные с DS компоненты:

```typescript
excludeUniqueLocalFromAdoption: true,
// Adoption = DS / (DS + shared-lib + переиспользуемые) × 100
// Уникальные (1 файл) из формулы убраны
```

## Вывод

DS Adoption Scanner закрывает конкретный gap: **измерение adoption дизайн-системы** как повторяемый, автоматизируемый процесс. Не требует SaaS, не требует изменений в продуктовых репозиториях, работает с любым набором пакетов.

Первый скан занимает 30 минут и даёт цифры, которых раньше не было.
